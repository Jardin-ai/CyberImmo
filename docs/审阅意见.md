  一、阻断性 Bug（上线前必须修复）

  1.1 生产必崩：RPC 名称不匹配

  文件：web/src/app/api/chat/route.ts:180
  问题：代码调用 admin.rpc("process_echo_transaction", {...})，但 migration_003_billing.sql 定义的函数名是 apply_billing_delta。
  每一次付费对话完成后，扣费调用必然报 function process_echo_transaction does not exist，导致用户消耗回声但余额不扣减，系统免费无限聊天。

  修复：将 route.ts 调用改为 apply_billing_delta，并对齐参数名（函数签名用 p_user_id/p_persona_id/p_delta/p_event_type/p_source_ref，而非
  route.ts 里传的 p_cost/p_action_type）。

  1.2 生产必崩：ChatClient 缺少 framer-motion 导入

  文件：web/src/app/chat/[id]/ChatClient.tsx:159
  问题：代码直接使用 AnimatePresence 和 motion，但文件顶部无对应 import。构建会通过（因为 @ts-ignore），但运行时立即崩溃 ReferenceError:
  AnimatePresence is not defined。

  修复：在文件顶部加 import { motion, AnimatePresence } from "framer-motion";

  1.3 日计数器非原子操作导致免费额度失效

  文件：web/src/app/api/chat/route.ts:190-195
  问题：dailyFreeTextUsed 在请求开始时读取，在流式响应结束后写回。高并发（哪怕 2 个用户同时发消息）时，两个请求读到相同的旧值，都执行
  +1，导致计数丢失。实际用了 10 次但计数只记录 5-6 次，FUP 形同虚设。
  修复：使用已有的 increment_daily_text_used RPC，替换应用层的 admin.from("users").update(...) 自增逻辑。

  1.4 Guest 数据同步的无限循环/重复创建

  文件：web/src/app/dashboard/DashboardContainer.tsx:33-75
  问题：useEffect 在每次 dashboard 挂载时检查 localStorage 并触发 submitOnboarding。如果网络慢、页面 re-render，或 window.location.reload()
  触发了新一轮挂载，会重复创建多个相同的 persona，每个都消耗免费 Token Grant。
  修复：写入 localStorage 一个 syncing 标志（或检查 persona 是否已存在），同步完成前阻止重复调用。建议改为 useRef 标志 + 服务端幂等检查。

  二、安全高危问题（上线前必须修复）

  2.1 RLS 策略允许任意用户写入计费记录

  文件：web/supabase/migrations/migration_001_initial.sql:131-139
  CREATE POLICY "billing_tokens_insert_service"
    ON public.billing_tokens FOR INSERT
    WITH CHECK (true);   -- ← 任何登录用户均可插入
  同样问题存在于 users_insert_service。这不是"服务角色策略"，只是"任何人都能插入"。任何人可以构造请求，给自己的账户插入 delta=999999
  的充值记录，或插入虚假用户行。
  修复：将策略改为 WITH CHECK (auth.role() = 'service_role') 或通过 SECURITY DEFINER RPC 统一写入，删除对 anon/authenticated role 的直接
  INSERT 授权。

  2.2 /api/chat 无速率限制，访客模式可被滥用

  文件：web/src/proxy.ts:8，web/src/app/api/chat/route.ts:42-70
  问题：/api/chat 在 PUBLIC_PATHS 中，访客模式完全绕过中间件。攻击者可以 isGuest=true 传任意 personaDraft，无限调用 ZhipuAI
  API，成本由产品方承担。目前无 IP 级速率限制、无每日上限。
  修复：至少在访客模式下限制每 IP 每日调用次数（如用 Supabase 边缘函数或 Cloudflare Rate Limiting）；或要求访客模式必须有有效的服务端
  token（而非客户端自报 isGuest=true）。

  2.3 Plan 要求的 Prompt Injection 拦截完全未实现

  文件：web/src/app/api/chat/route.ts
  计划 §7.1 第3步 明确要求关键词级 prompt injection 防护，代码中完全没有实现。产品核心场景（模拟已故亲人）极容易被攻击者用来提取 system
  prompt 或越界输出有害内容，存在舆情风险。

  2.4 Storage Bucket 权限不明确

  文件：ChatClient.tsx:121-132（chat-images bucket），SettingsClient.tsx:75（avatars bucket）
  Migration 中没有 Storage bucket 的 RLS 策略。不清楚这两个 bucket 是否为 public，若为 public，任何人可遍历所有用户上传的图片。

  三、功能缺失（影响产品可用性）

  3.1 支付是完整空壳，无任何真实支付集成

  文件：web/src/components/billing/PricingTable.tsx:125-129
  所有"立即开通"按钮触发 setToast("支付功能开发中，敬请期待 ✧")。没有 Stripe/微信支付/支付宝的任何接入代码，没有 Webhook
  handler，没有订阅激活逻辑。
  这是商业化 MVP 最核心的功能缺口。
  最低可行方案（手工闭环）：至少实现一个管理员手动充值界面（需要管理员密码），或提供一个带收款码的页面 + 运营通过 Supabase Dashboard 调用
  grant_subscription_echoes RPC。

  3.2 FREE 用户每日 quota 不会自动重置

  文件：web/supabase/migrations/migration_003_billing.sql:166-175
  pg_cron 任务被注释掉了。daily_free_text_used 字段永远不会清零，用完 5 次后 FREE 用户永久被封锁（除非手动 SQL 修复）。
  修复：在 Supabase Dashboard 启用 pg_cron 扩展，取消注释这段 SQL 并执行。

  3.3 开场白（Opening Message）从未注入对话

  文件：web/src/app/api/chat/route.ts:139-145
  问卷 Step5 收集了 openingMessage（用户最想说的话）和 aiFirstMessage（AI 开场白）。这两个字段进入了
  questionnaire_data，但在构建模型消息时从未被插入为初始 message，也未在聊天首屏展示。用户填了开场白，但对话从空屏开始，情感断层明显。

  3.4 System Prompt 每次请求重新构建，而非使用存储版本

  文件：web/src/app/api/chat/route.ts:139
  const systemPrompt = buildSystemPrompt(persona.questionnaire_data);
  但 personas 表已有 system_prompt TEXT NOT NULL 字段，onboarding 时已经构建并存储。现在每次对话都重新构建，不仅浪费计算，还可能导致 system
  prompt 内容与原始建档不一致（如果 buildSystemPrompt 函数版本升级后行为变化）。
  修复：直接使用 persona.system_prompt。

  3.5 历史消息上限设为 100，但超量时无分页

  文件：web/src/app/chat/[id]/page.tsx:36
  .limit(100) 硬编码，100 条以上的旧消息丢失。计划中的 Sliding Window 是"最近 20 条送入模型"，但 UI
  应该能完整展示历史。目前两者都没有正确实现：UI 最多 100 条，模型收到全量（没有 sliding window 裁切），Token 成本随对话变长线性增长。

  3.6 Subscription 过期检查缺失

  文件：web/src/app/api/chat/route.ts:97-101
  代码读取 subscription_tier 但从不检查 subscription_expires_at。订阅到期后用户仍享受付费权益（文字对话免费扣费），直到手动在数据库修改
  tier。

  3.7 导出对话 & 隐私保护 按钮是死按钮

  文件：web/src/components/MainSidebar.tsx:360-374
  两个功能按钮（导出对话记录、隐私保护设置）没有任何 onClick 处理逻辑，点击无反应。在 B2C
  情感类产品中，数据导出是用户信任的关键功能，缺失会影响付费意愿。

  四、架构与设计偏差

  4.1 模型选择与计划不符

  计划 §7.1：使用 DeepSeek OpenAI-compatible API
  实现：全部改为 ZhipuAI GLM-4/GLM-4-Flash
  这本身不是问题（可能是有意调整），但产品文案（PricingTable 的"DeepSeek 文字对话"）和实际模型不符，需要统一。同时 GLM-4 的中文情感表现与
  DeepSeek-V3 差异显著，应有对比测试。

  4.2 双余额字段产生语义混乱

  文件：web/src/lib/types.ts:10-11
  token_balance（legacy）和 echo_balance（canonical）同时存在，onboarding action 同时写两个字段，注释写着"legacy"但从未清理。migration_001
  只建了 token_balance，migration_003 加了 echo_balance 并做了迁移。这个技术债会持续产生混淆，尤其对于新接手的 Agent。

  4.3 Avatar 以 base64 存储在 JSONB，不可扩展

  文件：web/src/app/onboarding/steps/Step1Basic.tsx:46-67
  用户上传的 persona 头像经 compressImage 压缩后，作为 base64 字符串存入 personas.questionnaire_data.avatarUrl。一张头像可能 200-500KB，存在
   JSONB 字段里：① PostgreSQL 行体积膨胀；② 每次查询 persona 都传输整个 base64 串；③ 无法设 CDN 缓存；④ 聊天页面直接用 base64
  渲染，移动端首次加载慢。
  应上传至 Supabase Storage，存 URL。

  4.4 Session ID 复用 personaId，无法支持多会话

  文件：web/src/app/api/chat/route.ts:149
  const sessionId = personaId — 所有与同一 persona 的对话共享同一
  session，历史永不清零。如果用户想开启新对话（比如换个心情重新建立连接），无法实现。这在产品层面也会产生"千条消息历史"的加载问题。

  五、部署缺口（无法上线）

  5.1 无 Dockerfile 和 docker-compose.yml

  计划 §3、§10 Phase 5 要求 Dockerfile (Next.js standalone) + docker-compose.yml (单服务 web)，代码库中完全不存在。EC2 部署无从下手。

  5.2 无 .env.example，环境变量未文档化

  没有任何文件列出所需环境变量。新环境部署时需要猜测：
  - NEXT_PUBLIC_SUPABASE_URL
  - NEXT_PUBLIC_SUPABASE_ANON_KEY
  - SUPABASE_SERVICE_ROLE_KEY
  - ZHIPU_API_KEY（或 DEEPSEEK_API_KEY）

  5.3 Migration 执行状态不明

  Supabase 目录下有 5 个 migration 文件，但没有 Supabase CLI 的 config.toml，不清楚哪些 migration 已在目标数据库执行，哪些没有。同时
  migration_001 定义的 billing_tokens 约束不含 checkin，migration_003 改了这个约束——如果顺序不对或只执行了部分，会有静默数据错误。

  六、代码质量问题

  6.1 ChatClient 中大量 @ts-ignore

  文件：web/src/app/chat/[id]/ChatClient.tsx:62, 143
  使用 @ts-ignore 压制类型错误表明 AI SDK v6 类型与实际使用存在不匹配，应正确解决类型问题而非绕过。

  文件：web/src/app/dashboard/DashboardContainer.tsx:41, 52, 63

  6.3 Settings 中充值功能与 /billing 重复

  SettingsClient.tsx 里有 RECHARGE_OPTIONS 充值入口，BillingClient.tsx 里也有
  PricingTable。两处功能重叠但展示逻辑不同，维护成本加倍，且两处都是空壳，用户体验更混乱。

  6.4 auth/login 和 register 页面未检查

  根据目录存在，但未审阅其实现，需确认：邮箱 magic link 是否配置，是否有 redirect 参数的 open redirect 防护，注册后是否自动跳转 onboarding。

  七、UX/产品完整性

  7.1 聊天气泡无金色呼吸灯动效

  计划 §9.5 是产品核心差异化设计，最新 AI 消息应有 breathingGlow 动效。当前 ChatClient 对所有消息使用相同样式，最新消息没有任何特殊处理（无
  isLastAiMessage 判断，无动效）。

  7.2 Dashboard 列表未见 Persona 详情卡

  PersonaCard.tsx 文件存在，但 dashboard page 的具体渲染需确认是否正确显示 persona 卡片、最后对话时间、快速进入聊天入口。

  7.3 计划提到的 ModelSelector（模型切换）在聊天页不可见

  侧边栏规划了 ModelSelector（切换 DeepSeek-V3 / GLM-4.7-Flash），但当前聊天页 header
  没有此入口，用户无法感知自己在用哪个模型，也无法按需切换节省费用。

  7.4 注册后首次进入流程有断层风险

  用户注册 → 进入 dashboard → DashboardContainer 检测到 guest localStorage → 自动同步 → 成功后
  window.location.reload()。这整个同步发生在用户第一眼看到 dashboard 时，没有任何 loading UI，用户会看到一个闪白或内容突变，体验差。
